Product design specification · v1.0

Knoten. Automation you can read like a blueprint.

Knoten is a premium workflow automation platform: the canvas depth of n8n, the approachability of Zapier, the visual execution of Make — held together by the interaction standards of Figma, Linear, and Notion. This document is both the specification and a working demonstration: every color, component, and state below is rendered in the system it describes, in light and dark. Built to be implemented in React with Tailwind CSS.

n8n node canvas & data visibility Zapier progressive disclosure Make visible execution flow Figma direct manipulation Linear speed & keyboard-first Notion workflows as documents
01Foundations

Design philosophy

Automation tools fail in two directions: they become engineering consoles that intimidate operators, or wizards that collapse under real complexity. Knoten's position is a third one — a professional drafting instrument. Precise like an engineer's tool, legible like a document, fast like a native app. Five principles govern every screen.

From Figma — direct manipulation

The canvas is the product

Everything else — panels, dialogs, settings — exists in service of the canvas. Users build spatial memory of their workflows, so the canvas never rearranges itself without permission. Auto-layout is offered, never imposed. Selection, drag, and connection behave with pixel-perfect physicality: what you grab moves, what you point at highlights.

From Linear — speed as material

Sub-100ms or it didn't happen

Every interaction responds within 100ms — optimistically if the server hasn't answered. Full keyboard coverage: ⌘K reaches anything, single-key shortcuts operate the canvas. Loading is designed, not endured. Speed is treated as a design token as real as color.

From Zapier — progressive disclosure

Simple path first, power one level down

The first workflow takes five minutes without reading docs: pick a trigger, pick an action, done. Expression editors, error policies, and rate-limit config exist — one deliberate level below the surface, never removed, never in the way. Complexity is opt-in, not ambient.

From n8n & Make — honest data

Data is visible, always

A workflow is only trustworthy if you can see what flows through it. Every node shows its last real input and output. Test runs use live data, pinned safely. Mapping fields means dragging real values, not typing blind template strings. Nothing executes invisibly.

From Notion — the readable artifact

A workflow is a document

Workflows are read far more often than they are edited. Names, sticky notes, and group frames are first-class, so a colleague can understand a flow without running it. Every workflow has a description, an owner, and a change history — it's an artifact the team maintains, not a script someone once wrote.

Knoten's own rule

AI proposes, people commit

AI drafts workflows, explains failures, and suggests fixes — always as a visible, reviewable proposal rendered in the same node language the user edits by hand. AI output is never auto-committed, never a black box, and always cheaper to undo than to accept. Trust is the feature.

What "premium" means here

  • Density with calm. Operator-grade information density, but one accent color, one type family, generous whitespace between zones. The UI recedes; the user's workflow is the loudest thing on screen.
  • No dead ends. Every empty state teaches, every error proposes a next step, every limit explains itself. The product never shrugs.
  • Undo everything. Deleting a node, accepting an AI draft, editing a live workflow — all undoable via ⌘Z with a 30-step history. Versions snapshot on every publish.
02Foundations

Information architecture

The object model is a strict containment hierarchy — every screen answers "where am I?" with a breadcrumb of these objects. Navigation is two-level: a global rail for top-level areas, and the command palette (⌘K) as the universal shortcut across all of it.

ObjectContainsNotes
WorkspaceProjects, Members, Connections, SettingsBilling and identity boundary. One rail per workspace.
ProjectWorkflows, shared Data StoresTeam-level grouping (e.g. "Growth", "Ops"). Permissions attach here.
WorkflowVersions, Runs, VariablesThe document. Draft ↔ Published states; publishing snapshots a Version.
VersionNode graph (immutable)Every publish is diffable and restorable, like Figma version history.
RunSteps (one per executed node)An execution of one Version with concrete data. Retained per plan.
StepInput, Output, Logs, TimingThe atomic unit of debugging. Deep-linkable.

Global navigation

  • Rail (left, 240px, collapsible to 56px): Home · Workflows · Runs · Connections · Data Stores · Templates, with Members and Settings pinned to the bottom. Icon + label; collapsed state shows icons with tooltips.
  • Command palette (⌘K): fuzzy-searches workflows, runs, nodes, connections, settings pages, and actions ("Duplicate workflow", "Toggle theme"). Recency-ranked. This is the primary navigation for power users — the rail is the map, ⌘K is the teleporter.
  • Breadcrumb (top bar): Project / Workflow / (mode) — always visible in the builder and run views. Clicking any crumb navigates; the last crumb is an inline-editable title (Notion pattern).
  • Full-screen surfaces: the Builder and the Execution Viewer take over the content area but keep rail and breadcrumb, so orientation is never lost.

Behavior

  • Attention list is the hero. It ranks by (consecutive failures × workflow priority), not recency. A workspace with zero problems collapses the list into a quiet "All 142 workflows healthy" line with a checkmark — health is the default state, not an achievement banner.

  • Personalization: the dashboard is per-member. Recent workflows and pinned flows reflect the viewer, while health modules reflect the workspace.

Builder. The selected node gets the accent ring and its inspector; executed wires are green with the untaken branch left neutral — the last test run is readable directly from the graph. The bottom toolbar is Figma's grammar: tools, then objects, then AI.

Node anatomy

  • Card: 170–220px wide, icon tile (category-colored), name (user-editable), technical subtitle in mono. Corner badge shows last-run state: green check, red ✕, or nothing if never run.
  • Ports: 10px circles on the vertical midline; branch nodes stack labeled ports (true/false, case names). Dragging from a port draws a live wire; dropping on empty canvas opens the Node Library pre-filtered to compatible nodes — the highest-leverage shortcut in the product.
  • Wires: 2px cubic béziers. States: neutral (never run), accent (selected/hovered), green (data flowed in last run), dashed (disabled). Hovering a wire shows item count from the last run ("42 items").

06Surfaces

AI Builder

Invoked with ⌘J or the spark tool, the AI Builder turns a sentence into a reviewable draft. The contract is strict: AI output is always rendered as ghost nodes in the user's own node language — dashed, accent-tinted, inert — with a visible plan. Nothing runs, nothing saves, until the person accepts.

AI proposal state. Ghost nodes are dashed and accent-tinted; the plan card names every step, flags the missing Zendesk connection before acceptance, and offers exactly three verbs: Accept, Refine, Discard.

Interaction contract

  • Plan before graph. The numbered plan streams in first (readable in ~2s), then ghost nodes materialize step-by-step. Users can reject at the plan stage without ever seeing nodes.
  • Refine is conversational, scoped. "Make it hourly" edits the ghost draft in place — visible as a diff (changed ghosts pulse once). The thread stays attached to the draft until accepted or discarded.
  • Prerequisites surface early. Missing connections, scopes, or paid features are flagged in the plan, each with an inline fix action — never discovered at run time.
  • Accept = normal edit. Accepted nodes become ordinary draft nodes in one undo step. ⌘Z removes the whole draft.
  • Beyond building: the same surface explains failures ("why did run #8121 fail?") and drafts fixes as ghost edits — the error state (§18) links straight into it.

Execution Viewer. Duration bars share one time axis, so the 4.5-second CRM step visibly dwarfs everything else — the bottleneck is legible before reading a single number. The failed step carries a plain-language explanation with two concrete next actions.

Behavior

  • Live runs stream. An in-progress run animates top-to-bottom: pending steps are dimmed, the executing step pulses gently, durations tick in real time (Make's execution theater, but restrained).
  • Retry from step re-executes with the original upstream data — no need to re-trigger the webhook. Retried runs link back to their parent.
  • Data tools: item-by-item paging for batch runs, search within payloads, copy-as-JSON, and "pin as test data" to feed the builder.
  • Retention & redaction: payload retention is plan-based; fields marked sensitive in the schema render as ••• with per-field reveal, audited.
  • The runs index (workspace level) is Linear's issue list applied to runs: dense rows, saved filters ("Failed today", "Slower than 5s"), J/K navigation, ↩ to open.

Rules

  • Instant save for preferences, explicit save for danger. Toggles and selectors persist on change with a quiet "Saved" wisp. Anything destructive or billing-related gets an explicit confirm with the consequence spelled out.
  • Danger zone is a place, not a color scheme. Delete workspace / transfer ownership live at the bottom of General behind a divider, using the danger button variant. Typing the workspace name confirms deletion.
  • Connections page is operational, not just a list: each connection shows health (last used, error rate), which workflows depend on it, and re-auth in place — because expired auth is the #1 real-world failure.
  • Roles: Admin · Editor · Operator · Viewer. Operator can run, retry, and toggle workflows but not edit graphs — the on-call role other tools forget.

Knoten is honest about form factors: you build on a desktop, you monitor from anywhere. The canvas is not squeezed onto phones — instead, small screens get a first-class monitoring app: dashboards, runs, approvals, and the kill switch.

10System

Design system — "Blaupause"

The system is named Blaupause (blueprint). Three token layers, resolved in CSS custom properties so themes switch without a re-render, consumed in Tailwind via var()-backed utilities.

LayerExampleRule
1 · Primitivesblue-600: #2E4FD8 · slate-900: #171A26Raw scales. Never referenced by components.
2 · Semantic--surface, --ink-2, --accent, --err-softMeaning, not appearance. This layer is what themes remap — components below it never know which theme is active.
3 · Component--node-border-selected: var(--accent)Only where a component needs an override seam (nodes, canvas, charts).

Tailwind integration

  • Semantic tokens as Tailwind colors: colors: { surface: "var(--surface)", ink: { DEFAULT: "var(--ink)", 2: "var(--ink-2)", 3: "var(--ink-3)" }, accent: … } — so markup reads bg-surface text-ink-2 border-line and is automatically theme-correct.
  • Theming: tokens defined on :root, remapped under @media (prefers-color-scheme: dark) and explicitly via [data-theme="dark"] / [data-theme="light"] — user choice always beats OS preference. (This document uses exactly that mechanism.)
  • Component library: Radix primitives (or React Aria) + CVA variants + Tailwind. Canvas rendered on react-flow-class custom engine with DOM nodes and SVG wires, virtualized beyond 200 nodes.
  • One source of truth: tokens live in tokens.css; Tailwind config only references, never redefines. Charts read the same custom properties, so dataviz can't drift from UI.

Rules

  • Validated: both theme palettes pass the four-check colorblind audit (lightness band, chroma floor, CVD adjacent-pair ΔE ≥ 12, ≥3:1 contrast against their surface) — light set #2E4FD8 / #188A4C / #B45309 / #C93838, dark set #5E77E8 / #2FAE6E / #C08327 / #D95560. Dark is its own selection, not an inversion.
  • Status is never color-alone: always paired with an icon or label (pill dots + words everywhere in this spec).
  • Text wears ink tokens only. Colored text is limited to status pills, links, and the accent — body copy never borrows chart or status colors.
  • Charts: single-series uses accent; success/failure splits use ok/err with 2px surface gaps between stacked segments. Categorical series beyond that draw from the fixed order accent → ok → warn → err, never cycled hues.
✓ Do

  • Let neutral surfaces carry 95% of every screen
✕ Don't
  • Introduce per-integration brand colors into the UI chrome
  • Use green for "new" or "featured" — it means succeeded
  • Put accent text on accent-soft in dark mode without checking contrast

16System

Animations

Motion communicates causality, never personality. Budget: if it doesn't explain where something came from, what changed, or what's alive, it doesn't animate. Enter fast, exit faster; nothing bounces except a wire snapping into a port.

ease-out-quick150ms · cubic-bezier(.2,.8,.3,1)Hovers, toggles, selection rings

  • Canvas motion is physics, not choreography: pan/zoom are direct 1:1; only ⌘0 fit animates (240ms) so users keep spatial context.
  • List reordering uses FLIP transforms so filtered items visibly move rather than teleport.
  • Reduced motion: prefers-reduced-motion collapses everything to opacity fades ≤100ms and stops loops (this page already obeys it). Live-run state falls back to a static badge swap.
  • No skeleton dance: shimmer is subtle, one direction, and never on more than one region at once.

Loading states

Loading is tiered by duration, and mutations never wait at all — Linear's optimistic model. Skeletons mirror the real layout exactly so nothing jumps when content lands.

DurationTreatment
< 100msNothing. A flash of spinner is worse than a beat of silence.
100–400msLocal shimmer only in the region that's loading — the shell never blanks.
> 400msSkeleton matching final layout (below). Row count matches the expected data.
> 5s (jobs)Progress with real counts ("Importing 1,204 of 3,000 rows"), cancellable, continues in background with a toast on completion.
MutationsOptimistic: toggle flips, node moves, name edits apply instantly and reconcile; on failure they revert with a toast explaining why.

Accessibility guidelines

Target: WCAG 2.2 AA everywhere, including the canvas — the part most node editors write off. Accessibility review is part of component definition-of-done, not a later audit.

The keyboard-complete canvas

  • The graph is navigable without a pointer: the canvas is a focusable application region; Tab enters it, arrow keys move focus along wires (→ downstream, ↑↓ between siblings), ↩ opens the inspector, Esc returns to the canvas.
  • Every pointer action has a keyboard twin: connect = C then pick target from an ordered list; move = arrows (8px) / ⇧-arrows (32px); marquee = ⇧-extend selection.
  • A parallel list view (toggle: ⌘⇧L) renders the same workflow as an ordered outline — the screen-reader-native representation, and the "table view" equivalent for the graph.

Contrast & color

  • Text ≥ 4.5:1 (ink-3 on surface is tuned to pass at its sizes); UI boundaries and focus indicators ≥ 3:1; both themes validated independently — dark is its own palette, not an inversion.
  • Status is always icon + label, never a bare dot (chart marks get gaps and direct labels); the palettes above pass CVD adjacent-pair checks by construction.
  • Focus: 2px accent ring, 2px offset, on every interactive element including ports and wires; focus is never outline: none'd without a replacement.

Live & assistive behavior

  • Run streaming announces politely: one aria-live="polite" region summarizes ("Step 3 of 5 succeeded, 204 milliseconds") instead of firing per-frame updates.
  • Toasts are role="status", pause their timer on hover/focus, and every toast action is reachable before dismissal.
  • Motion & vestibular safety: prefers-reduced-motion removes pans, zoom animations, shimmer, and pulse loops product-wide.
  • Targets: minimum 24×24px effective hit area (44px on touch); tooltips are hoverable per WCAG 1.4.13.
  • Testing gate: axe-core in CI, keyboard-only walkthrough of every new surface, and VoiceOver + NVDA passes on the four core flows (build, run, debug, fix) each release.
